# Hook/工具类型配置

## Research 必读文件

按优先级排序——先理解调用方式，再理解契约，最后深入实现：

1. `__tests__/*.test.ts(x)`（测试用例）— 最先读，理解输入/输出/边界行为
2. 导出入口（`index.ts` 或被其他模块导入的文件）— 确认公共 API 边界
3. 实现文件（`.ts` / `.tsx`）— 理解内部逻辑
4. 调用方代码（grep 导入语句，抽样 2-3 个典型调用方）— 理解实际使用模式

## 复杂度分级

```text
是否包含状态机、并发控制、或复杂的副作用编排？
├─ 是 → 复杂
└─ 否 → 是否有 3+ 个配置维度，或需要管理内部状态/副作用？
         ├─ 是 → 中等
         └─ 否 → 简单
```

| 级别 | 约束条数参考 | 典型目标 |
|------|-----------|---------| 
| 简单 | 3–6 条 | 纯函数工具（cn, filterDOMProps） |
| 中等 | 6–12 条 | 有状态的 hook（useControllableState, useOverlay） |
| 复杂 | 12+ 条 | 状态机/并发 hook（useAsyncList, useVirtualizer） |

## 逆向工程示例表

| 痕迹类型 | ❌ 逆向工程写法 | ✅ 设计文档写法 |
|---------|----------------|---------------|
| 引用内部数据结构 | "内部用 WeakMap 缓存实例" | "对相同输入保证引用稳定" |
| 描述具体实现机制 | "通过 setTimeout 实现 debounce" | "支持防抖，避免高频触发" |
| 引用内部变量名 | "维护 `_prevValue` 和 `_nextValue` 两个引用" | "需要追踪值的变化方向以决定过渡效果" |
| 描述具体的错误处理代码 | "catch 块中调用 retry() 并递减 retryCount" | "失败时自动重试，支持配置最大重试次数" |
| 描述闭包技巧 | "通过 useRef 保存最新的 callback 避免闭包陈旧" | "回调始终引用最新版本，不受闭包捕获影响" |
| 描述类型体操 | "通过 `Extract<T, string>` 约束泛型参数" | "仅接受字符串类型的键" |

## Verify 额外过滤项

子 agent 校验时，除通用过滤规则外，还需额外过滤：

- 内部数据结构选择（如 Map vs WeakMap vs Object）
- 具体的缓存/记忆化实现方式
- 闭包/引用管理的具体技巧
- 类型体操和泛型约束的具体写法

## 内容取舍补充

应该写：
- 输入/输出的语义契约
- 副作用的触发条件和清理策略
- 引用稳定性保证

不应该写：
- 内部数据结构选择（Map vs Object 等）
- 闭包/引用管理的具体技巧
- 类型体操的具体写法
