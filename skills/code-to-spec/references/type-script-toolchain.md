# 脚本/工具链类型配置

## Research 必读文件

按优先级排序——先理解用途和配置，再理解执行流程：

1. `README.md` 或文件头部注释 — 理解脚本用途和使用方式
2. 配置文件 / schema 定义（如 `build.config.ts`、`ci.json`）— 理解配置维度
3. 主入口文件 — 理解执行流程和阶段划分
4. 插件/扩展点（如有）— 理解可扩展性设计

## 复杂度分级

```text
是否有多阶段 pipeline、插件系统、或复杂的错误恢复机制？
├─ 是 → 复杂
└─ 否 → 是否有可配置项、多种运行模式、或扩展点？
         ├─ 是 → 中等
         └─ 否 → 简单
```

| 级别 | 约束条数参考 | 典型目标 |
|------|-----------|---------| 
| 简单 | 3–6 条 | 单一流程脚本（cleanup.js） |
| 中等 | 6–12 条 | 可配置的构建脚本（gen-tokens, build-entry） |
| 复杂 | 12+ 条 | 多阶段 pipeline（style-dictionary build, MCP server） |

## 逆向工程示例表

| 痕迹类型 | ❌ 逆向工程写法 | ✅ 设计文档写法 |
|---------|----------------|---------------|
| 描述文件操作细节 | "用 `fs.readFileSync` 读取配置后 `JSON.parse`" | "从配置文件加载运行参数" |
| 描述 shell 命令拼接 | "拼接 `git log --format='%H' \| head -n 1`" | "获取最近一次提交的标识" |
| 引用临时文件路径 | "写入 `/tmp/.cache/build-${hash}.json`" | "构建过程支持增量缓存以加速重复构建" |
| 描述正则表达式 | "用 `/^export\s+(default\s+)?/` 匹配导出语句" | "解析模块的导出声明" |
| 描述错误处理的具体分支 | "ENOENT 时创建目录，EACCES 时抛出权限错误" | "优雅处理目标路径不存在和权限不足的情况" |

## Verify 额外过滤项

子 agent 校验时，除通用过滤规则外，还需额外过滤：

- 具体的文件系统操作方式（如 fs.readFile vs fs.createReadStream）
- shell 命令拼接和转义细节
- 临时文件/目录的路径和清理方式
- 正则表达式的具体写法

## 内容取舍补充

应该写：
- pipeline 阶段划分和依赖关系
- 配置 schema 的设计意图
- 幂等性和可重入性要求

不应该写：
- 文件系统操作的具体 API 选择
- 正则表达式的具体写法
- 临时文件的路径和清理方式
