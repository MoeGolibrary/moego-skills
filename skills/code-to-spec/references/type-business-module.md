# 业务模块类型配置

## Research 必读文件

按优先级排序——先理解业务场景，再理解技术边界：

1. `README.md` 或模块文档 — 理解业务背景和职责
2. 入口文件 / 路由定义 / 服务注册 — 确认模块边界和对外接口
3. 核心逻辑文件 — 理解主要业务流程
4. 依赖服务的接口定义（类型文件、API client）— 理解外部依赖

## 复杂度分级

```text
是否涉及事务/saga、复杂状态流转、或多服务编排？
├─ 是 → 复杂
└─ 否 → 是否有多个子服务协调、或 3 种以上业务状态？
         ├─ 是 → 中等
         └─ 否 → 简单
```

| 级别 | 约束条数参考 | 典型目标 |
|------|-----------|---------| 
| 简单 | 3–6 条 | 单一职责 CRUD 模块 |
| 中等 | 6–12 条 | 多服务协调模块 |
| 复杂 | 12+ 条 | 事务编排、复杂状态机模块 |

## 章节模板

文件位置：模块根目录。

```markdown
# Name 规格

本文档描述 Name 模块的业务职责、技术边界与关键设计约束，作为实现的核心参考。

## 技术集成场景
## 关键设计约束

## 不适用场景
```

## 逆向工程示例表

| 痕迹类型 | ❌ 逆向工程写法 | ✅ 设计文档写法 |
|---------|----------------|---------------|
| 描述数据库查询 | "通过 `SELECT ... JOIN` 关联用户表和订单表" | "需要关联用户信息和订单数据" |
| 描述 HTTP 请求构造 | "POST 到 `/api/v2/analytics` 并附带 `X-Request-ID` header" | "向分析服务上报事件数据，需要支持请求追踪" |
| 描述序列化细节 | "用 `JSON.stringify` 后 `Buffer.from` 转为 base64" | "数据需要序列化为可传输格式" |
| 描述具体的重试策略代码 | "指数退避，初始 100ms，最大 5000ms，jitter 0.5" | "网络请求失败时采用指数退避重试策略" |

## Verify 额外过滤项

子 agent 校验时，除通用过滤规则外，还需额外过滤：

- 具体的 ORM/数据库查询写法
- HTTP 请求的具体构造方式
- 序列化/反序列化的具体格式处理
- 日志/监控的具体埋点方式

## 内容取舍补充

应该写：
- 领域模型和聚合边界
- 事务边界和一致性要求
- 权限模型和访问控制策略

不应该写：
- 具体的数据库查询语句
- HTTP 请求的具体构造方式
- 日志/监控的具体埋点代码
