# SPEC 各章节写作指南

本指南按模块类型提供各章节的写作要点。所有类型共享"关键设计约束"、"适用场景"、"不适用场景"三个章节的写作规则。

---

## UI 组件

### 终端用户场景

- 句式："作为终端用户，我可以……"
- 每个场景用三级标题分组（如"层级浏览与选择"、"键盘操作"）
- 聚焦用户可感知的功能和交互，不涉及实现细节
- 必须覆盖：核心交互、状态反馈（加载/禁用/选中等）、键盘操作（如有）、无障碍访问

### 开发者场景

- 句式："作为开发者，我可以……"
- 每个场景用三级标题分组
- 覆盖 API 能力：数据接入、受控/非受控、样式自定义（classNames slots 列表）、事件回调、ref 方法等
- 列出关键 props 和配置项，但不需要穷举所有 props
- 多种使用模式（如静态声明 vs 动态渲染）需分别说明
- 只描述公共 API（通过 index.ts 导出的内容）；未导出的内部模块和 hook 不应作为开发者场景出现

---

## Hook/工具

### 调用者场景

- 句式："作为调用者，我可以……"
- 每个场景用三级标题分组（如"基础调用"、"高级配置"、"错误处理"）
- 聚焦调用者视角的能力：输入什么、得到什么、在什么条件下行为会变化
- 必须覆盖：基本用法、配置选项、边界行为（空输入/异常输入）、与其他 hook/工具的组合使用
- 对于 hook：说明返回值的语义和更新时机
- 对于工具函数：说明纯函数/副作用特性、是否幂等

### API 契约

- 描述输入参数的语义和约束（不是类型定义，而是"为什么需要这个参数"）
- 描述输出/返回值的语义保证（如"引用稳定"、"惰性计算"）
- 描述副作用（如有）：触发条件、清理时机
- 描述错误行为：什么情况下抛出/返回错误、错误的语义
- 如果有多个重载或模式，分别说明各模式的适用条件

---

## 脚本/工具链

### 使用者场景

- 句式："作为使用者，我可以……"
- 每个场景用三级标题分组（如"基本运行"、"自定义配置"、"CI 集成"）
- 聚焦使用者视角：怎么运行、运行后得到什么、在什么条件下行为会变化
- 必须覆盖：基本运行方式、输入/输出（文件/stdout/副作用）、环境要求
- 如果有多种运行模式（如 dev/prod/watch），分别说明

### 配置与扩展

- 描述配置项的语义和设计意图（不是配置文件的 schema 文档）
- 描述扩展点（如有）：插件接口、自定义 transform、hook 点
- 描述环境变量和外部依赖的要求
- 如果有多阶段 pipeline，描述各阶段的职责和依赖关系

---

## 业务模块

### 业务场景

- 句式："在……场景下，系统需要……"
- 每个场景用三级标题分组（如"核心流程"、"异常处理"、"权限控制"）
- 从业务视角描述模块解决的问题和提供的能力
- 必须覆盖：核心业务流程、业务规则和约束、异常/边界场景
- 避免技术术语，用业务语言描述

### 技术集成场景

- 句式："作为集成方，我可以……"
- 每个场景用三级标题分组
- 从技术视角描述模块的对外接口和集成方式
- 覆盖：服务接口、事件/消息、依赖注入点、配置方式
- 描述与其他模块的协作关系和数据流向

## 通用章节

以下章节的写作规则对所有模块类型一致。

### 关键设计约束

这是 SPEC 中最重要的部分。

复杂模块可使用子标题分组（如"模块架构"、"状态管理"、"核心算法"等），具体分组策略参见 SKILL.md 中的"复杂度分级"。

每条约束应简洁明确，说清"是什么"和"为什么"（如果原因非显而易见）。

逐条写作、逐条检查，禁止批量写完再统一检查。逆向工程过滤规则和自检清单见 SKILL.md 的"约束过滤原则"章节。

### 适用场景

- 列举 3-5 个典型使用场景
- 简短描述即可

### 不适用场景

- 列举替代方案，格式："场景描述 → 使用 `替代方案`"
- 帮助使用者快速判断是否应该使用此模块
